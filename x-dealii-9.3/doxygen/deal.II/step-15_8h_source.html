564  * <h4>MinimalSurfaceProblem::setup_system</h4>
565  *
566 
567  *
568  * As always in the setup-system function, we setup the variables of the
569  * finite element method. There are same differences to @ref step_6 "step-6", because
570  * there we start solving the PDE from scratch in every refinement cycle
571  * whereas here we need to take the solution from the previous mesh onto the
572  * current mesh. Consequently, we can't just reset solution vectors. The
573  * argument passed to this function thus indicates whether we can
574  * distributed degrees of freedom (plus compute constraints) and set the
575  * solution vector to zero or whether this has happened elsewhere already
576  * (specifically, in <code>refine_mesh()</code>).
577  *
578 
579  *
580  *
581  * @code

588  * current_solution.reinit(dof_handler.n_dofs());

590  * hanging_node_constraints.clear();
591  * DoFTools::make_hanging_node_constraints(dof_handler,
592  * hanging_node_constraints);
593  * hanging_node_constraints.close();
594  * }

597  * @endcode

620  * <a name="MinimalSurfaceProblemassemble_system"></a>
621  * <h4>MinimalSurfaceProblem::assemble_system</h4>
622  *
623 
624  *
625  * This function does the same as in the previous tutorials except that now,
626  * of course, the matrix and right hand side functions depend on the
627  * previous iteration's solution. As discussed in the introduction, we need
628  * to use zero boundary values for the Newton updates; we compute them at
629  * the end of this function.
630  *
631 
632  *
633  * The top of the function contains the usual boilerplate code, setting up
634  * the objects that allow us to evaluate shape functions at quadrature
635  * points and temporary storage locations for the local matrices and
636  * vectors, as well as for the gradients of the previous solution at the
637  * quadrature points. We then start the loop over all cells:
638  *
639  * @code
640  * template <int dim>
641  * void MinimalSurfaceProblem<dim>::assemble_system()
642  * {
643  * const QGauss<dim> quadrature_formula(fe.degree + 1);
644  *
645  * system_matrix = 0;
646  * system_rhs = 0;
647  *
648  * FEValues<dim> fe_values(fe,
649  * quadrature_formula,
650  * update_gradients | update_quadrature_points |
651  * update_JxW_values);
652  *
653  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
654  * const unsigned int n_q_points = quadrature_formula.size();
655  *
656  * FullMatrix<double> cell_matrix(dofs_per_cell, dofs_per_cell);
657  * Vector<double> cell_rhs(dofs_per_cell);
658  *
659  * std::vector<Tensor<1, dim>> old_solution_gradients(n_q_points);
660  *
661  * std::vector<types::global_dof_index> local_dof_indices(dofs_per_cell);
662  *
663  * for (const auto &cell : dof_handler.active_cell_iterators())
664  * {
665  * cell_matrix = 0;
666  * cell_rhs = 0;
667  *
668  * fe_values.reinit(cell);
669  *
670  * @endcode
671  *
672  * For the assembly of the linear system, we have to obtain the values
673  * of the previous solution's gradients at the quadrature
674  * points. There is a standard way of doing this: the
675  * FEValues::get_function_gradients function takes a vector that
676  * represents a finite element field defined on a DoFHandler, and
677  * evaluates the gradients of this field at the quadrature points of the
678  * cell with which the FEValues object has last been reinitialized.
679  * The values of the gradients at all quadrature points are then written
680  * into the second argument:
681  *
682  * @code
683  * fe_values.get_function_gradients(current_solution,
684  * old_solution_gradients);
685  *
686  * @endcode
687  *
688  * With this, we can then do the integration loop over all quadrature
689  * points and shape functions. Having just computed the gradients of
690  * the old solution in the quadrature points, we are able to compute
691  * the coefficients @f$a_{n}@f$ in these points. The assembly of the
692  * system itself then looks similar to what we always do with the
693  * exception of the nonlinear terms, as does copying the results from
694  * the local objects into the global ones:
695  *
696  * @code
697  * for (unsigned int q = 0; q < n_q_points; ++q)
698  * {
699  * const double coeff =
700  * 1.0 / std::sqrt(1 + old_solution_gradients[q] *
701  * old_solution_gradients[q]);
702  *
703  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
704  * {
705  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
706  * cell_matrix(i, j) +=
707  * (((fe_values.shape_grad(i, q) // ((\nabla \phi_i
708  * * coeff // * a_n
709  * * fe_values.shape_grad(j, q)) // * \nabla \phi_j)
710  * - // -
711  * (fe_values.shape_grad(i, q) // (\nabla \phi_i
712  * * coeff * coeff * coeff // * a_n^3
713  * * (fe_values.shape_grad(j, q) // * (\nabla \phi_j
714  * * old_solution_gradients[q]) // * \nabla u_n)
715  * * old_solution_gradients[q])) // * \nabla u_n)))
716  * * fe_values.JxW(q)); // * dx
717  *
718  * cell_rhs(i) -= (fe_values.shape_grad(i, q) // \nabla \phi_i
719  * * coeff // * a_n
720  * * old_solution_gradients[q] // * u_n
721  * * fe_values.JxW(q)); // * dx
722  * }
723  * }
724  *
725  * cell->get_dof_indices(local_dof_indices);
726  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
727  * {
728  * for (unsigned int j = 0; j < dofs_per_cell; ++j)
729  * system_matrix.add(local_dof_indices[i],
730  * local_dof_indices[j],
731  * cell_matrix(i, j));
732  *
733  * system_rhs(local_dof_indices[i]) += cell_rhs(i);
734  * }
735  * }
736  *
737  * @endcode
738  *
739  * Finally, we remove hanging nodes from the system and apply zero
740  * boundary values to the linear system that defines the Newton updates
741  * @f$\delta u^n@f$:
742  *
743  * @code
744  * hanging_node_constraints.condense(system_matrix);
745  * hanging_node_constraints.condense(system_rhs);
746  *
747  * std::map<types::global_dof_index, double> boundary_values;
748  * VectorTools::interpolate_boundary_values(dof_handler,
749  * 0,
750  * Functions::ZeroFunction<dim>(),
751  * boundary_values);
752  * MatrixTools::apply_boundary_values(boundary_values,
753  * system_matrix,
754  * newton_update,
755  * system_rhs);
756  * }
757  *
758  *
759  *
760  * @endcode

773  * template <int dim>

786  *

795  * <a name="MinimalSurfaceProblemrefine_mesh"></a>
796  * <h4>MinimalSurfaceProblem::refine_mesh</h4>
797  *
798 
799  *
800  * The first part of this function is the same as in @ref step_6 "step-6"... However,
801  * after refining the mesh we have to transfer the old solution to the new
802  * one which we do with the help of the SolutionTransfer class. The process
803  * is slightly convoluted, so let us describe it in detail:
804  *
805  * @code
806  * template <int dim>
807  * void MinimalSurfaceProblem<dim>::refine_mesh()
808  * {
809  * Vector<float> estimated_error_per_cell(triangulation.n_active_cells());
810  *
811  * KellyErrorEstimator<dim>::estimate(
812  * dof_handler,
813  * QGauss<dim - 1>(fe.degree + 1),
814  * std::map<types::boundary_id, const Function<dim> *>(),
815  * current_solution,
816  * estimated_error_per_cell);
817  *
818  * GridRefinement::refine_and_coarsen_fixed_number(triangulation,
819  * estimated_error_per_cell,
820  * 0.3,
821  * 0.03);
822  *
823  * @endcode
824  *
825  * Then we need an additional step: if, for example, you flag a cell that
826  * is once more refined than its neighbor, and that neighbor is not
827  * flagged for refinement, we would end up with a jump of two refinement
828  * levels across a cell interface. To avoid these situations, the library
829  * will silently also have to refine the neighbor cell once. It does so by
830  * calling the Triangulation::prepare_coarsening_and_refinement function
831  * before actually doing the refinement and coarsening. This function
832  * flags a set of additional cells for refinement or coarsening, to
833  * enforce rules like the one-hanging-node rule. The cells that are
834  * flagged for refinement and coarsening after calling this function are
835  * exactly the ones that will actually be refined or coarsened. Usually,
836  * you don't have to do this by hand
837  * (Triangulation::execute_coarsening_and_refinement does this for
838  * you). However, we need to initialize the SolutionTransfer class and it
839  * needs to know the final set of cells that will be coarsened or refined
840  * in order to store the data from the old mesh and transfer to the new
841  * one. Thus, we call the function by hand:
842  *
843  * @code
844  * triangulation.prepare_coarsening_and_refinement();
845  *
846  * @endcode
847  *
848  * With this out of the way, we initialize a SolutionTransfer object with
849  * the present DoFHandler and attach the solution vector to it, followed
850  * by doing the actual refinement and distribution of degrees of freedom
851  * on the new mesh
852  *
853  * @code
854  * SolutionTransfer<dim> solution_transfer(dof_handler);
855  * solution_transfer.prepare_for_coarsening_and_refinement(current_solution);
856  *
857  * triangulation.execute_coarsening_and_refinement();
858  *
859  * dof_handler.distribute_dofs(fe);
860  *
861  * @endcode
862  *
863  * Finally, we retrieve the old solution interpolated to the new
864  * mesh. Since the SolutionTransfer function does not actually store the
865  * values of the old solution, but rather indices, we need to preserve the
866  * old solution vector until we have gotten the new interpolated
867  * values. Thus, we have the new values written into a temporary vector,
868  * and only afterwards write them into the solution vector object. Once we
869  * have this solution we have to make sure that the @f$u^n@f$ we now have
870  * actually has the correct boundary values. As explained at the end of
871  * the introduction, this is not automatically the case even if the
872  * solution before refinement had the correct boundary values, and so we
873  * have to explicitly make sure that it now has:
874  *
875  * @code
876  * Vector<double> tmp(dof_handler.n_dofs());
877  * solution_transfer.interpolate(current_solution, tmp);
878  * current_solution = tmp;
879  *
880  * set_boundary_values();
881  *
882  * @endcode
883  *
884  * On the new mesh, there are different hanging nodes, which we have to
885  * compute again. To ensure there are no hanging nodes of the old mesh in
886  * the object, it's first cleared. To be on the safe side, we then also
887  * make sure that the current solution's vector entries satisfy the
888  * hanging node constraints (see the discussion in the documentation of
889  * the SolutionTransfer class for why this is necessary):
890  *
891  * @code
892  * hanging_node_constraints.clear();
893  *
894  * DoFTools::make_hanging_node_constraints(dof_handler,
895  * hanging_node_constraints);
896  * hanging_node_constraints.close();
897  *
898  * hanging_node_constraints.distribute(current_solution);
899  *
900  * @endcode
901  *
902  * We end the function by updating all the remaining data structures,
903  * indicating to <code>setup_dofs()</code> that this is not the first
904  * go-around and that it needs to preserve the content of the solution
905  * vector:
906  *
907  * @code
908  * setup_system(false);
909  * }
910  *
911  *
912  *
913  * @endcode
914  *
915  *
916  * <a name="MinimalSurfaceProblemset_boundary_values"></a>
917  * <h4>MinimalSurfaceProblem::set_boundary_values</h4>
918  *
919 
920  *
921  * The next function ensures that the solution vector's entries respect the
922  * boundary values for our problem. Having refined the mesh (or just
923  * started computations), there might be new nodal points on the
924  * boundary. These have values that are simply interpolated from the
925  * previous mesh (or are just zero), instead of the correct boundary
926  * values. This is fixed up by setting all boundary nodes explicit to the
927  * right value:
928  *
929  * @code
930  * template <int dim>
931  * void MinimalSurfaceProblem<dim>::set_boundary_values()
932  * {
933  * std::map<types::global_dof_index, double> boundary_values;
934  * VectorTools::interpolate_boundary_values(dof_handler,
935  * 0,
936  * BoundaryValues<dim>(),
937  * boundary_values);
938  * for (auto &boundary_value : boundary_values)
939  * current_solution(boundary_value.first) = boundary_value.second;
940  * }
941  *
942  *
943  * @endcode
944  *
945  *
946  * <a name="MinimalSurfaceProblemcompute_residual"></a>
947  * <h4>MinimalSurfaceProblem::compute_residual</h4>
948  *
949 
950  *
951  * In order to monitor convergence, we need a way to compute the norm of the
952  * (discrete) residual, i.e., the norm of the vector
953  * @f$\left<F(u^n),\varphi_i\right>@f$ with @f$F(u)=-\nabla \cdot \left(
954  * \frac{1}{\sqrt{1+|\nabla u|^{2}}}\nabla u \right)@f$ as discussed in the
955  * introduction. It turns out that (although we don't use this feature in
956  * the current version of the program) one needs to compute the residual
957  * @f$\left<F(u^n+\alpha^n\;\delta u^n),\varphi_i\right>@f$ when determining
958  * optimal step lengths, and so this is what we implement here: the function
959  * takes the step length @f$\alpha^n@f$ as an argument. The original
960  * functionality is of course obtained by passing a zero as argument.
961  *
962 
963  *
964  * In the function below, we first set up a vector for the residual, and
965  * then a vector for the evaluation point @f$u^n+\alpha^n\;\delta u^n@f$. This
966  * is followed by the same boilerplate code we use for all integration
967  * operations:
968  *
969  * @code
970  * template <int dim>
971  * double MinimalSurfaceProblem<dim>::compute_residual(const double alpha) const
972  * {
973  * Vector<double> residual(dof_handler.n_dofs());
974  *
975  * Vector<double> evaluation_point(dof_handler.n_dofs());
976  * evaluation_point = current_solution;
977  * evaluation_point.add(alpha, newton_update);
978  *
979  * const QGauss<dim> quadrature_formula(fe.degree + 1);
980  * FEValues<dim> fe_values(fe,
981  * quadrature_formula,
982  * update_gradients | update_quadrature_points |
983  * update_JxW_values);
984  *
985  * const unsigned int dofs_per_cell = fe.n_dofs_per_cell();
986  * const unsigned int n_q_points = quadrature_formula.size();
987  *
988  * Vector<double> cell_residual(dofs_per_cell);
989  * std::vector<Tensor<1, dim>> gradients(n_q_points);
990  *
991  * std::vector<types::global_dof_index> local_dof_indices(dofs_per_cell);
992  *
993  * for (const auto &cell : dof_handler.active_cell_iterators())
994  * {
995  * cell_residual = 0;
996  * fe_values.reinit(cell);
997  *
998  * @endcode
999  *
1000  * The actual computation is much as in
1001  * <code>assemble_system()</code>. We first evaluate the gradients of
1002  * @f$u^n+\alpha^n\,\delta u^n@f$ at the quadrature points, then compute
1003  * the coefficient @f$a_n@f$, and then plug it all into the formula for
1004  * the residual:
1005  *
1006  * @code
1007  * fe_values.get_function_gradients(evaluation_point, gradients);
1008  *
1009  *
1010  * for (unsigned int q = 0; q < n_q_points; ++q)
1011  * {
1012  * const double coeff =
1013  * 1. / std::sqrt(1 + gradients[q] * gradients[q]);
1014  *
1015  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1016  * cell_residual(i) -= (fe_values.shape_grad(i, q) // \nabla \phi_i
1017  * * coeff // * a_n
1018  * * gradients[q] // * u_n
1019  * * fe_values.JxW(q)); // * dx
1020  * }
1021  *
1022  * cell->get_dof_indices(local_dof_indices);
1023  * for (unsigned int i = 0; i < dofs_per_cell; ++i)
1024  * residual(local_dof_indices[i]) += cell_residual(i);
1025  * }
1026  *
1027  * @endcode
1028  *
1029  * At the end of this function we also have to deal with the hanging node
1030  * constraints and with the issue of boundary values. With regard to the
1031  * latter, we have to set to zero the elements of the residual vector for
1032  * all entries that correspond to degrees of freedom that sit at the
1033  * boundary. The reason is that because the value of the solution there is
1034  * fixed, they are of course no "real" degrees of freedom and so, strictly
1035  * speaking, we shouldn't have assembled entries in the residual vector
1036  * for them. However, as we always do, we want to do exactly the same
1037  * thing on every cell and so we didn't not want to deal with the question
1038  * of whether a particular degree of freedom sits at the boundary in the
1039  * integration above. Rather, we will simply set to zero these entries
1040  * after the fact. To this end, we need to determine which degrees
1041  * of freedom do in fact belong to the boundary and then loop over all of
1042  * those and set the residual entry to zero. This happens in the following
1043  * lines which we have already seen used in @ref step_11 "step-11", using the appropriate
1044  * function from namespace DoFTools:
1045  *
1046  * @code
1047  * hanging_node_constraints.condense(residual);
1048  *
1049  * for (types::global_dof_index i :
1050  * DoFTools::extract_boundary_dofs(dof_handler))
1051  * residual(i) = 0;
1052  *
1053  * @endcode
1054  *
1055  * At the end of the function, we return the norm of the residual:
1056  *
1057  * @code
1058  * return residual.l2_norm();
1059  * }
1060  *
1061  *
1062  *
1063  * @endcode
1064  *
1065  *
1066  * <a name="MinimalSurfaceProblemdetermine_step_length"></a>
1067  * <h4>MinimalSurfaceProblem::determine_step_length</h4>
1068  *
1069 
1070  *
1071  * As discussed in the introduction, Newton's method frequently does not
1072  * converge if we always take full steps, i.e., compute @f$u^{n+1}=u^n+\delta
1073  * u^n@f$. Rather, one needs a damping parameter (step length) @f$\alpha^n@f$ and
1074  * set @f$u^{n+1}=u^n+\alpha^n\delta u^n@f$. This function is the one called
1075  * to compute @f$\alpha^n@f$.
1076  *
1077 
1078  *
1079  * Here, we simply always return 0.1. This is of course a sub-optimal
1080  * choice: ideally, what one wants is that the step size goes to one as we
1081  * get closer to the solution, so that we get to enjoy the rapid quadratic
1082  * convergence of Newton's method. We will discuss better strategies below
1083  * in the results section.
1084  *
1085  * @code
1086  * template <int dim>
1087  * double MinimalSurfaceProblem<dim>::determine_step_length() const
1088  * {
1089  * return 0.1;
1090  * }
1091  *
1092  *
1093  *
1094  * @endcode
1095  *
1096  *
1097  * <a name="MinimalSurfaceProblemoutput_results"></a>
1098  * <h4>MinimalSurfaceProblem::output_results</h4>
1099  *
1100 
1101  *
1102  * This last function to be called from `run()` outputs the current solution
1103  * (and the Newton update) in graphical form as a VTU file. It is entirely the
1104  * same as what has been used in previous tutorials.
1105  *
1106  * @code
1107  * template <int dim>
1108  * void MinimalSurfaceProblem<dim>::output_results(
1109  * const unsigned int refinement_cycle) const
1110  * {
1111  * DataOut<dim> data_out;
1112  *
1113  * data_out.attach_dof_handler(dof_handler);
1114  * data_out.add_data_vector(current_solution, "solution");
1115  * data_out.add_data_vector(newton_update, "update");
1116  * data_out.build_patches();
1117  *
1118  * const std::string filename =
1119  * "solution-" + Utilities::int_to_string(refinement_cycle, 2) + ".vtu";
1120  * std::ofstream output(filename);
1121  * data_out.write_vtu(output);

1371 <a name="Possibilitiesforextensions"></a><h3>Possibilities for extensions</h3>
1372 
1373 
1374 The program shows the basic structure of a solver for a nonlinear, stationary
1375 problem. However, it does not converge particularly fast, for good reasons:
1376 
1377 - The program always takes a step size of 0.1. This precludes the rapid,
1378  quadratic convergence for which Newton's method is typically chosen.
1379 - It does not connect the nonlinear iteration with the mesh refinement
1380  iteration.
1381 
1382 Obviously, a better program would have to address these two points.
1383 We will discuss them in the following.
1384 
1385 
1386 <a name="Steplengthcontrol"></a><h4> Step length control </h4>
1387 
1388 
1389 Newton's method has two well known properties:
1390 - It may not converge from arbitrarily chosen starting points. Rather, a
1391  starting point has to be close enough to the solution to guarantee
1392  convergence. However, we can enlarge the area from which Newton's method
1393  converges by damping the iteration using a <i>step length</i> 0<@f$\alpha^n\le
1394  1@f$.
1395 - It exhibits rapid convergence of quadratic order if (i) the step length is
1396  chosen as @f$\alpha^n=1@f$, and (ii) it does in fact converge with this choice
1397  of step length.
1398 
1399 A consequence of these two observations is that a successful strategy is to
1400 choose @f$\alpha^n<1@f$ for the initial iterations until the iterate has come
1401 close enough to allow for convergence with full step length, at which point we
1402 want to switch to @f$\alpha^n=1@f$. The question is how to choose @f$\alpha^n@f$ in an
1403 automatic fashion that satisfies these criteria.
1404 
1405 We do not want to review the literature on this topic here, but only briefly
1406 mention that there are two fundamental approaches to the problem: backtracking
1407 line search and trust region methods. The former is more widely used for
1408 partial differential equations and essentially does the following:
1409 - Compute a search direction
1410 - See if the resulting residual of @f$u^n + \alpha^n\;\delta u^n@f$ with
1411  @f$\alpha^n=1@f$ is "substantially smaller" than that of @f$u^n@f$ alone.
1412 - If so, then take @f$\alpha^n=1@f$.
1413 - If not, try whether the residual is "substantially smaller" with
1414  @f$\alpha^n=2/3@f$.
1415 - If so, then take @f$\alpha^n=2/3@f$.
1416 - If not, try whether the residual is "substantially smaller" with
1417  @f$\alpha^n=(2/3)^2@f$.
1418 - Etc.
1419 One can of course choose other factors @f$r, r^2, \ldots@f$ than the @f$2/3,
1420 (2/3)^2, \ldots@f$ chosen above, for @f$0<r<1@f$. It is obvious where the term
1421 "backtracking" comes from: we try a long step, but if that doesn't work we try
1422 a shorter step, and ever shorter step, etc. The function
1423 <code>determine_step_length()</code> is written the way it is to support
1424 exactly this kind of use case.
1425 
1426 Whether we accept a particular step length @f$\alpha^n@f$ depends on how we define
1427 "substantially smaller". There are a number of ways to do so, but without
1428 going into detail let us just mention that the most common ones are to use the
1429 Wolfe and Armijo-Goldstein conditions. For these, one can show the following:
1430 - There is always a step length @f$\alpha^n@f$ for which the conditions are
1431  satisfied, i.e., the iteration never gets stuck as long as the problem is
1432  convex.
1433 - If we are close enough to the solution, then the conditions allow for
1434  @f$\alpha^n=1@f$, thereby enabling quadratic convergence.
1435 
1436 We will not dwell on this here any further but leave the implementation of
1437 such algorithms as an exercise. We note, however, that when implemented
1438 correctly then it is a common observation that most reasonably nonlinear
1439 problems can be solved in anywhere between 5 and 15 Newton iterations to
1440 engineering accuracy &mdash; substantially fewer than we need with the current
1441 version of the program.
1442 
1443 More details on globalization methods including backtracking can be found,
1444 for example, in @cite GNS08 and @cite NW99.
1445 
1446 A separate point, very much worthwhile making, however, is that in practice
1447 the implementation of efficient nonlinear solvers is about as complicated as
1448 the implementation of efficient finite element methods. One should not
1449 attempt to reinvent the wheel by implementing all of the necessary steps
1450 oneself. Rather, just like building finite element solvers on libraries
1451 such as deal.II, one should be building nonlinear solvers on libraries such
1452 as [SUNDIALS](https://computing.llnl.gov/projects/sundials). In fact,
1453 deal.II has interfaces to SUNDIALS and in particular to its nonlinear solver
1454 sub-package KINSOL through the SUNDIALS::KINSOL class. It would not be
1455 very difficult to base the current problem on that interface.

1459 <a name="Integratingmeshrefinementandnonlinearandlinearsolvers"></a><h4> Integrating mesh refinement and nonlinear and linear solvers </h4>
1460 
1461 
1462 We currently do exactly 5 iterations on each mesh. But is this optimal? One
1463 could ask the following questions:
1464 - Maybe it is worthwhile doing more iterations on the initial meshes since
1465  there, computations are cheap.
1466 - On the other hand, we do not want to do too many iterations on every mesh:
1467  yes, we could drive the residual to zero on every mesh, but that would only
1468  mean that the nonlinear iteration error is far smaller than the
1469  discretization error.
1470 - Should we use solve the linear systems in each Newton step with higher or
1471  lower accuracy?
1472 
1473 Ultimately, what this boils down to is that we somehow need to couple the
1474 discretization error on the current mesh with the nonlinear residual we want
1475 to achieve with the Newton iterations on a given mesh, and to the linear
1476 iteration we want to achieve with the CG method within each Newton
1477 iterations.
1478 
1479 How to do this is, again, not entirely trivial, and we again leave it as a
1480 future exercise.
